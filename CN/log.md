- [新建] [[search-r1]] + [[search-r1-codebase-walkthrough]] + [[agentic-rl-foundations]] —— 三页一起做，从 Shiki 一整 session 关于 Search-R1 当 agentic-RL 入门的深度交流里整出来的。**(1) [[search-r1]]** 是标准论文精读页，覆盖 Jin et al., COLM 2025（arXiv:2503.09516）。Search-R1 把 DeepSeek-R1-Zero 从纯推理扩展到 tool use：outcome-only EM reward + 标准 PPO/GRPO + 4-tag 协议（`<think>/<search>/<information>/<answer>`）+ load-bearing 的 **retrieved-token loss masking**（把梯度挡在环境注入 token 之外）。标志数：Qwen2.5-7B-base + PPO 在 7 个 QA 数据集（NQ/TriviaQA/PopQA/HotpotQA/2wiki/Musique/Bamboogle）上达到 0.431 EM 平均，比 RAG +42%，比 rejection sampling +24%，比纯 RL 无搜索 +56%。关键消融：无 mask 0.431 → 0.343（-25% 相对）。PPO 在多轮长 rollout 上打败 GRPO（vs DeepSeek-R1 的 GRPO 偏好反直觉）。我的批评：只测单工具 QA、EM 是噪声 reward、max_turns=2 偏小、没 30B+ 规模、输出格式脆弱（提取要求 ≥2 个 `<answer>` match）。**(2) [[search-r1-codebase-walkthrough]]** 是完整代码教程 —— repo 共 ~33K 行 Python 但 Search-R1 自己的贡献只有 ~600 行论文代码 + ray_trainer.py 15 行补丁，跑在标准 veRL PPO/GRPO 机器上。走 Part A（search_r1/ —— train_ppo.sh、nq_search.py、retrieval_server.py、generation.py 含 run_llm_loop / execute_predictions / _info_masked_concatenate_with_padding 深读、tensor_helper.py、infer.py）和 Part B（veRL —— main_ppo.py 含 RewardManager、rl_dataset.py、ray_trainer.py:fit() 含 apply_kl_penalty + compute_advantage、core_algos.py 含 GAE / GRPO / PPO clipped loss / 4 种 KL 估计、dp_actor.py:update_policy、dp_critic.py、vllm_rollout.py 含 hybrid engine、fsdp_workers.py）。加上一次训练 step 的端到端调用栈、文件级归属表、5 难度上手实验建议。**(3) [[agentic-rl-foundations]]** 是入门 hub：4 阶段阅读路径（RL 基础 → R1-Zero 谱系 → Search-R1 → 生产基础设施 → 前沿）、标志参考迷你摘要、8 条 FAQ（包括 "agentic RL 用 PPO 还是 GRPO"、"什么时候用 OPD 代替 RL"、"retrieved-token masking 实际在做什么" 等）。加 9 条 2026+ 开放研究方向。Search-R1 的 4-tag 协议和 Table 3 的 PPO-vs-GRPO 反直觉结果现在是一等 wiki 内容。强交叉链接到 [[prorl-agent]]、[[polar]]、[[nemo-gym]]、[[grpo]]、[[ppo-for-llm]]、[[on-policy-distillation]]。顶层和 agentic-rl/index.md 都更新。EN + CN。citation 文件已建。

- [摄入] arXiv:2605.24220 "Polar: Agentic RL on Any Harness at Scale"（Xu, Zhang, Zhang, Han, Liu, Hu, Diao, Jin, Zou, Demoret, Kautz, Dong；NVIDIA；2026-05-22）—— 论文精读在 [[polar]]，位于 `wiki/agentic-rl/`。**[[prorl-agent|ProRL Agent]] 的直接续作** —— 同一 NVIDIA 团队（作者重合度 ~75 %）、同一 GitHub repo（`NVIDIA-NeMo/ProRL-Agent-Server`，原地重写）。名字来自 "Pr**O**rL Agent serv**R**"。架构转变：把 ProRL Agent 的 `AgentHandler` ABC plugin 模型换成 **LLM-API proxy**，proxy 坐在未修改 agent harness 和 inference server 之间，拦截 Anthropic Messages / OpenAI Chat / OpenAI Responses / Google generateContent 流量，记 token IDs + log-probs，在 harness 之外重建 token-faithful 轨迹。可训练 harness 集合从"NVIDIA-curated 小列表"扩到"任何打 LLM API 的"—— Codex、Claude Code、Qwen Code、Pi、Gemini CLI、OpenCode。两种轨迹策略：`per_request`（每次 model call = 一条 trace，baseline，一个 session 上千碎片）vs `prefix_merging`（按严格 token 前缀关系链化 completion，屏蔽 canonical interstitial，保留 behavior-policy token 可训练，子 agent / 上下文压缩 / prompt 改写天然成独立链）。标志数：Qwen3.5-4B + simple GRPO 在 SkyRL-v0-293-data，SWE-Bench Verified —— Codex 3.8 → 26.4 (+22.6 pp)、Claude Code 29.8 → 34.6 (+4.8)、Qwen Code 34.6 → 35.2 (+0.6)、Pi 34.2 → 40.4 (+6.2)。关键消融：同 workload，`per_request` = 1185 trainer 更新 / 189.5 min / 20.4 % rollout GPU 利用率；`prefix_merging` = 218 更新 / 35.2 min / 87.7 % 利用率 —— **wall-clock 5.39× 加速，利用率 4.3× 提升**。per-request + outcome reward 广播会触发严重 reward hacking。**Polar 已注册为 NeMo Gym 环境** —— 这就是昨天 [[prorl-agent#ProRL Agent vs NeMo Gym —— 同族、不同层]] 节里说"还没有公开桥"的那座桥；那条预测错了（或者说桥同一周就在路上）。发布 SFT 语料：`nvidia/polar-swegym-pi-qwen35-122b-a10b-trajectories` 在 HF（Apache-2.0、504 条通过 SWE-Bench 的轨迹、64 GPU 小时）。我的批评：所有实验都是 4B（没扫模型规模）；都是 coding 任务（声明"any harness"但没测 web/OS/scientific harness）；prefix_merging 在 harness 重写 prompt 时退化为 per_request；interstitial token 代价对上下文重的 harness 是真问题；per-request reward hacking 留给 future work；没跟 ProRL Agent 一对一对照；provider API 稳定性依赖厂商不敌对（Anthropic / OpenAI 可能故意打破 proxy 替换）。在 ProRL Agent 页面顶加了显眼的"superseded"警告 callout，修正了它"adapter 不存在"那段。NeMo Gym 相关阅读也更新了。EN + CN。

- [摄入] arXiv:2604.15039 "Prefill-as-a-Service: KVCache of Next-Generation Models Could Go Cross-Datacenter"（Qin, He, Wang, Li, Xu, Wu, Zheng, Zhang；Moonshot AI + 清华；v1 2026-04-16 / v2 2026-04-22）—— 论文精读在 [[prfaas]]，位于 `wiki/llm-inference/`。跟 Mooncake 同一位清华主导（Mingxing Zhang）—— PrfaaS 是 Mooncake 的跨数据中心续作。核心论点：hybrid attention 模型（Kimi Linear、MiMo-V2-Flash、Qwen3.5-397B、Ring-2.5-1T）相比 dense GQA 把每实例 KV 吞吐砍了约 13×，把 PD 的可部署网络边界从 RDMA 级 fabric 推到跨 DC 的普通以太网。但单靠 hybrid attention 不够 —— burst、长度分布偏斜、cache 局部性、DC 间带宽抖动都咬人。PrfaaS-PD 加三件系统侧的事：(1) 长度阈值选择性外放（$l > t$ → PrfaaS，否则本地 PD-P），(2) 混合 prefix cache 池统一管理 full-attention block KV 和 linear-attention recurrent state，区分 prefix-cache / transfer-cache 块，(3) 双时间尺度调度器 —— 短期带宽+cache 感知路由，长期 Np/Nd 重分配。吞吐模型 Λ_max = min(Θ_prfaas/p, Θ_pd-p/(1-p), Θ_pd-d)，两个优化旋钮（阈值 t、PD 集群拆分）。1T Kimi-Linear 案例研究（32 H200 PrfaaS + 64 H20 本地 PD vs 96 H20 同构）：吞吐 +54 %，P90 TTFT −64 %，13 Gbps 出口 = 100 Gbps 以太网的 13 %，等成本下 ~15 % 增益。Naive 异构（无调度）只有 1.16× —— 调度器贡献 ~25 % 吞吐增益，这是 load-bearing 对照。我的批评：是案例研究不是部署（所有数字来自稳态解析模型 + profiling，不是真跑）；输出长度死定 1024（长输出推理负载会改变最优）；无公开源码；跨 DC 链路丢包的故障模式没说；隐私 / 多租户完全沉默；对 dense GQA 模型完全无帮助（架构整体押注 hybrid attention 行业方向）；没跟 [[af-disaggregation|AF 分离]] 正面对照。在 [[prefill-decode-disaggregation]] 的"前沿方向"加了"跨数据中心 PD"小节作交叉链接；[[af-disaggregation]] 相关阅读也加了。EN + CN 都做了。

- [摄入] arXiv:2605.22675 "Self-Policy Distillation via Capability-Selective Subspace Projection"（Hao, Shang, Long, Zhao, Liang；剑桥/港科/芝大；2026-05-21 预印本）—— 论文精读在 [[self-policy-distillation]]，位于 `wiki/rl-infra/`。无老师、无 RL 的自蒸馏路线：用一个 20-500 例校准集的 **correctness-aligned NLL** 梯度做 SVD 提取低秩能力子空间，在中间层 + 最后一层构造投影矩阵 $P_K^{(\ell)}, P_V^{(\ell)}$，自生成时挂为 hook 把 KV 激活投影到能力子空间，再在引导后的输出上做标准 LoRA SFT。标志数（Qwen2.5-0.5B）：vs Base 平均 +8.9 %，vs SSD +6.4 %；OOD 迁移 —— QA 校准的子空间把 GSM8K 11→26 %、MBPP 17→21 %。关键消融：full-sequence loss 在 MBPP 上*比 Base 还差*（11.9 %），correctness-aligned 拿到 25.5 % —— token-mask 选择是 load-bearing 的。我的批评：小模型 showcase（涨幅随规模衰减）；correctness-token 规则藏在 App. A.1；正文没给 rank $r$；"self-policy" 名字是 RL 营销（没有真正的策略梯度）。在 [[on-policy-distillation]] 的变体分类表里新增一行。EN + CN 都做了。
