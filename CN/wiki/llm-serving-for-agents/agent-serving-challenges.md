# AI 智能体的独特服务挑战

## 概述

为 AI 智能体提供 LLM 推理服务与传统聊天机器人服务有着**根本性的区别**。智能体工作负载的特征是：

- **多次 LLM 调用**：每个任务 5-50+ 次 LLM 调用，中间穿插工具执行
- **不可预测的计算模式**：上下文长度动态增长，任务时长从秒级到小时级
- **长时间运行的会话**：需要跨多轮维护 KV 缓存状态
- **严格的端到端 SLA**：用户关心整个任务的完成时间，而非单次响应延迟

这些特征对传统的 LLM 服务架构提出了全新的挑战。

```
传统聊天服务:
  用户 ──[请求]──> LLM ──[响应]──> 用户
  (一次往返, ~1-3秒)

智能体服务:
  用户 ──[任务]──> LLM ──[调用工具]──> 工具
                    ^                    │
                    │    [工具结果]       │
                    └────────────────────┘
                    (重复 5-50 次, 总计 30秒-10分钟+)
```

## 智能体 vs. 聊天机器人工作负载

### 详细对比

| 维度 | 聊天机器人 | 智能体 |
|------|-----------|--------|
| **每任务 LLM 调用数** | 1 | 5-50+ |
| **上下文增长模式** | 线性（用户消息累积） | 非线性（工具结果爆发式累积） |
| **延迟关注点** | 单次响应的 TTFT/TPOT | 端到端任务完成时间 |
| **会话持续时间** | 秒-分钟 | 分钟-小时 |
| **状态管理** | 基于会话 | 多轮 + 暂停/恢复 |
| **前缀复用** | 高（系统提示不变） | 混合（工具结果改变上下文） |
| **输出格式** | 自由文本 | 结构化（JSON、函数调用） |
| **失败恢复** | 简单重试 | 需要检查点和回滚 |
| **资源使用** | 可预测 | 高度不可预测 |
| **并发模式** | 大量短连接 | 少量长连接 |

### 典型工作负载对比

```
聊天机器人 (单轮):
  Token 消耗: ~500 (输入) + ~200 (输出) = ~700 tokens
  持续时间: ~2 秒
  GPU 占用: ~2 秒

编码智能体 (多轮):
  Turn 1:  系统提示(2000) + 用户请求(100) → 输出(500) + 工具调用
  Turn 2:  +上轮(2600) + 工具结果(3000)   → 输出(300) + 工具调用
  Turn 3:  +上轮(5900) + 工具结果(2000)   → 输出(400) + 工具调用
  ...
  Turn 10: +上轮(25000) + 工具结果(1500)  → 输出(800) = 最终答案

  Token 消耗: ~27,300 (输入累积) + ~4,500 (输出) = ~31,800 tokens
  持续时间: ~2 分钟
  GPU 占用: 断续，穿插工具执行等待

上下文增长曲线:
  Tokens
  │
30k│                              ●
25k│                         ●
20k│                    ●
15k│               ●
10k│          ●
 5k│     ●
 2k│●
   └──────────────────────────────
    T1   T2   T3   T4   T5  ... T10

  注意: 这是累积输入 token 数，不含输出
  每轮的预填充成本递增，无优化则总成本 = O(n^2)
```

## 关键挑战详解

### 1. 多轮 KV 缓存管理

**问题**：智能体会话在 LLM 推理和工具执行之间交替。工具执行期间（可能持续数秒到数分钟），LLM 处于空闲状态，但其 KV 缓存需要被保留以供下一轮使用。

```
时间线:
  t0        t1       t2        t3       t4        t5
  │─ LLM ──│─ 工具 ──│── LLM ──│─ 工具 ──│── LLM ──│
  │ 推理    │ 执行    │  推理   │  执行   │  推理    │
  │         │ (5秒)   │        │ (30秒)  │         │
  │         │         │        │         │         │

KV 缓存状态:
  ████████  ????????  ████████  ????????  ████████
  (使用中)  (闲置但   (使用中)  (闲置但   (使用中)
            需保留)             需保留)

  如果在工具执行期间驱逐 KV 缓存:
  - 下一轮需要重新预填充整个上下文
  - Turn 5 重新预填充 25,000+ tokens
  - 延迟增加数秒，GPU 浪费严重

  如果保留所有 KV 缓存:
  - GPU 内存被大量闲置缓存占用
  - 减少可服务的并发请求数
  - 内存碎片化
```

**核心矛盾**：保留缓存占用宝贵的 GPU 内存；驱逐缓存导致昂贵的重计算。

**解决方案**：

| 方案 | 机制 | 效果 |
|------|------|------|
| **Continuum TTL** | 基于重新加载成本设置 KV 缓存 TTL | 延迟降低 1.12x-3.66x |
| **LMCache** | 将 KV 缓存卸载到 CPU/磁盘/S3 | 吞吐量提升 15x |
| **KV 缓存感知调度** | 将请求路由到有热缓存的节点 | 减少缓存未命中 |
| **分层存储** | GPU → CPU → 磁盘 → S3 | 平衡成本和性能 |

### 2. 上下文窗口压力

**问题**：每次工具调用都会向上下文添加结果，10 步的智能体可能累积数千甚至数万 tokens。

```
上下文窗口使用情况:

  ┌──────────────────────────────────────┐ 128K 上限
  │                                      │
  │  空闲空间 (逐渐减少)                  │
  │                                      │
  ├──────────────────────────────────────┤ ← 工具结果不断填充
  │  Turn 10 工具结果 (1,500)            │
  ├──────────────────────────────────────┤
  │  Turn 9 工具结果 (2,000)             │
  ├──────────────────────────────────────┤
  │  ...                                 │
  ├──────────────────────────────────────┤
  │  Turn 1 工具结果 (3,000)             │
  ├──────────────────────────────────────┤
  │  系统提示 + 工具定义 (2,000)          │
  └──────────────────────────────────────┘

  后果:
  - 接近上限时性能退化 ("lost in the middle")
  - 超出限制时需要截断，丢失历史信息
  - 更长的上下文 = 更多 prefill 计算 = 更高延迟
```

**缓解策略**：
- **上下文压缩**：对旧的工具结果进行摘要
- **滑动窗口**：只保留最近 N 轮的完整信息
- **选择性保留**：只保留与当前子任务相关的历史
- **分层摘要**：距离越远的历史，摘要越精简

详见 [[multi-turn-optimization]]。

### 3. 延迟组合

**问题**：智能体的端到端延迟是所有 LLM 调用和工具执行的累加。

```
延迟分解:

  总延迟 = Σ(LLM 延迟) + Σ(工具延迟) + Σ(网络开销)

  典型 10 步智能体:
  ┌─────┬──────┬─────┬──────┬─────┬──────┬─────┐
  │LLM  │ Tool │ LLM │ Tool │ LLM │ Tool │ ... │
  │2.5s │ 1.0s │ 3.0s│ 5.0s │ 2.0s│ 0.5s │     │
  └─────┴──────┴─────┴──────┴─────┴──────┴─────┘

  10 步 × (平均 2.5s LLM + 平均 2.0s 工具) = ~45 秒
  + 网络开销 ~5 秒
  = 总计 ~50 秒

  延迟优化目标:
  1. 减少 LLM TTFT (通过 KV 缓存复用)
  2. 减少工具执行时间 (并行化、缓存)
  3. 减少 LLM 调用次数 (更好的规划)
  4. 流式传输中间结果

  每步 TTFT 的影响:
  ┌──────────────────────────────────────┐
  │ 无缓存: TTFT = 2.5s × 10 = 25s     │
  │ 前缀缓存: TTFT = 0.3s × 10 = 3s    │
  │ 节省: 22s (88% TTFT 降低)           │
  └──────────────────────────────────────┘
```

### 4. 提示缓存悖论

**来源**："Don't Break the Cache" (2026)

```
直觉上: 缓存整个智能体上下文应该最优

实际结果:
  ┌──────────────────────────────────────────┐
  │  策略           │ 成本节省 │ TTFT 改善   │
  ├──────────────────────────────────────────┤
  │  不缓存         │  0%     │  0%         │
  │  缓存整个上下文  │  20-40% │  有时更差!   │
  │  仅缓存系统提示  │  45-80% │  13-31%     │
  └──────────────────────────────────────────┘

  原因分析:
  1. 智能体的每轮上下文都不同（工具结果不同）
  2. 全上下文缓存的命中率很低
  3. 缓存管理开销反而增加了延迟
  4. 系统提示在所有轮次中完全相同 → 命中率 100%

  最佳策略:
  - 系统提示 + 工具定义 → 始终缓存 (稳定前缀)
  - 对话历史 → 使用 KV 缓存复用 (前缀匹配)
  - 工具结果 → 不缓存提示缓存，依赖 KV 缓存
```

### 5. 不可预测的调度

**问题**：传统 LLM 服务假设请求是相对均匀的，但智能体工作负载高度不规则。

```
传统聊天负载:
  请求/秒
  │  ────────────────────────────
  │  (相对稳定的请求率)
  └──────────────────────────────> 时间

智能体负载:
  请求/秒
  │    ▲         ▲▲
  │    ││        │││    ▲
  │    │││       ││││   ││
  │  ▲ │││  ▲   │││││  ││  ▲
  │──│──│──│──│──│──│──│──│───> 时间
  │  突发   休息  突发    突发

  特征:
  - 工具执行期间: 0 个请求到 LLM
  - 工具完成后: 突发请求
  - 上下文长度: 每轮递增
  - 同一个会话的请求必须路由到同一节点 (缓存亲和性)
```

**调度挑战**：

| 挑战 | 影响 | 解决方向 |
|------|------|---------|
| 突发请求模式 | 峰值负载难以预测 | 弹性扩缩容 |
| 缓存亲和性 | 需要 sticky session | KV 缓存感知路由 |
| 异构上下文长度 | 不同请求资源需求差异大 | 分桶调度 |
| 长任务阻塞 | 长运行任务占用资源 | 抢占式调度 |
| 工具等待空隙 | GPU 在工具执行时空闲 | 间隙复用 (Continuum) |

### 6. 可靠性要求

**问题**：智能体任务可能运行数分钟甚至数小时。单次失败可能导致整个任务重新开始。

```
可靠性需求:

  10 步智能体, 每步成功率 p:
  整体成功率 = p^10

  p=99%  → 90.4%  (可接受)
  p=95%  → 59.9%  (勉强)
  p=90%  → 34.9%  (不可接受)

  必须实现:
  ┌────────────────────────────────────────┐
  │  1. 检查点 (Checkpointing)             │
  │     - 每步完成后保存状态                │
  │     - 失败后从最近检查点恢复            │
  │                                        │
  │  2. 重试策略                           │
  │     - 指数退避重试                     │
  │     - 区分可重试/不可重试错误           │
  │                                        │
  │  3. 优雅降级                           │
  │     - 工具失败时使用备选方案            │
  │     - 模型不可用时切换备选模型          │
  │                                        │
  │  4. 超时管理                           │
  │     - 每步设置合理超时                  │
  │     - 总任务时间限制                    │
  └────────────────────────────────────────┘
```

```python
# 可靠性增强的智能体服务配置
class AgentServingConfig:
    # 重试配置
    max_retries: int = 3
    retry_backoff_base: float = 1.5  # 秒
    retry_backoff_max: float = 30.0

    # 超时配置
    llm_call_timeout: float = 60.0   # 单次 LLM 调用超时
    tool_call_timeout: float = 120.0  # 工具调用超时
    total_task_timeout: float = 600.0 # 整个任务超时

    # 检查点配置
    checkpoint_enabled: bool = True
    checkpoint_interval: int = 1      # 每步检查点
    checkpoint_storage: str = "redis"

    # KV 缓存配置
    kv_cache_ttl: float = 300.0      # 5 分钟 TTL
    kv_cache_offload: bool = True    # 允许卸载到 CPU
```

### 7. 成本管理

**问题**：智能体消耗的 token 数量远超聊天机器人，且难以预测。

```
成本对比 (以 Claude 3.5 Sonnet 为例):

  聊天机器人:
  - 输入: 500 tokens × $3/M = $0.0015
  - 输出: 200 tokens × $15/M = $0.003
  - 总计: ~$0.0045/对话

  编码智能体 (10 步):
  - 累积输入: ~30,000 tokens × $3/M = $0.09
  - 累积输出: ~5,000 tokens × $15/M = $0.075
  - 总计: ~$0.165/任务

  比率: 智能体 ≈ 37x 聊天机器人

  企业级成本 (每天 10,000 个智能体任务):
  - 智能体: $1,650/天 = ~$50,000/月
  - 聊天机器人: $45/天 = ~$1,350/月

成本优化策略:
  ┌──────────────────────────────────────┐
  │ 1. 前缀缓存: 减少 50-80% 输入成本   │
  │ 2. 模型级联: 简单步骤用便宜模型       │
  │ 3. 上下文压缩: 减少累积 token 数      │
  │ 4. 规划优化: 减少不必要的步骤         │
  │ 5. 批处理: 合并独立的工具调用         │
  └──────────────────────────────────────┘
```

### 8. 多模型编排

**问题**：复合 AI 系统中，不同组件可能使用不同的模型，增加了编排复杂度。

```
典型编码智能体的模型编排:

  ┌─────────┐    ┌────────────────┐
  │ 路由器   │───>│ 简单查询:      │
  │ (小模型) │    │ Haiku/GPT-4o-mini│
  └────┬────┘    └────────────────┘
       │
       │         ┌────────────────┐
       ├────────>│ 复杂推理:      │
       │         │ Opus/o3        │
       │         └────────────────┘
       │
       │         ┌────────────────┐
       └────────>│ 代码生成:      │
                 │ Sonnet/GPT-4o  │
                 └────────────────┘

  编排挑战:
  - 不同模型的 API 格式不同
  - 延迟特性不同
  - KV 缓存不能跨模型共享
  - 故障隔离和降级策略
```

## 专用系统

### Continuum (arXiv 2511.02230)

**核心创新**：为智能体工作负载设计的 KV 缓存 TTL 机制。

```
Continuum 架构:

  ┌─────────────────────────────────────┐
  │           Continuum 调度器           │
  │                                     │
  │  ┌─────────┐    ┌──────────────┐   │
  │  │ TTL 管理 │    │ 重新加载成本  │   │
  │  │          │    │ 估算器       │   │
  │  └────┬────┘    └──────┬───────┘   │
  │       │                │            │
  │       v                v            │
  │  ┌──────────────────────────┐      │
  │  │   KV 缓存池              │      │
  │  │   [会话A: TTL=30s] ████  │      │
  │  │   [会话B: TTL=5s]  ██    │      │
  │  │   [会话C: 已过期] ░░░   │      │
  │  └──────────────────────────┘      │
  └─────────────────────────────────────┘

工作原理:
1. LLM 推理完成，返回工具调用
2. 估算该会话的重新加载成本 (基于上下文长度)
3. 设置 TTL = f(reload_cost, 内存压力)
4. 工具执行期间锁定 KV 缓存
5. 工具结果返回，继续推理 (无需重新预填充)
```

**性能**：
- 延迟降低 **1.12x-3.66x**
- 吞吐量提升 **1.10x-3.22x**

### Pie (SOSP 2025)

可编程推理引擎，为不同的 "inferlet"（推理单元）提供不同的服务策略。

### KVFlow

工作流感知的 KV 缓存驱逐，使用 Agent Step Graph 建模智能体工作流，预测哪些缓存条目最可能被复用。

## 服务配置示例

### 面向智能体的 vLLM 配置

```python
# vLLM 智能体优化配置
from vllm import LLM, SamplingParams

llm = LLM(
    model="anthropic/claude-3-sonnet",
    # KV 缓存配置
    enable_prefix_caching=True,          # 启用前缀缓存
    gpu_memory_utilization=0.85,         # 为缓存留出空间
    max_model_len=131072,                # 支持长上下文
    # 调度配置
    max_num_seqs=32,                     # 并发请求数
    max_num_batched_tokens=65536,        # 批处理 token 上限
    # 智能体优化
    enable_chunked_prefill=True,         # 分块预填充
    preemption_mode="recompute",         # 抢占模式
)

# 智能体请求处理
async def handle_agent_request(session_id: str, messages: list):
    """处理智能体请求，利用会话亲和性"""
    sampling_params = SamplingParams(
        temperature=0.0,       # 确定性输出
        max_tokens=4096,       # 允许长输出
        stop=["</tool_call>"], # 工具调用停止符
    )

    # 利用前缀缓存: 相同 session_id 的请求
    # 自动匹配之前的 KV 缓存
    result = await llm.generate(
        messages,
        sampling_params,
        # 会话追踪
        request_id=f"{session_id}_{turn_number}"
    )
    return result
```

### 智能体服务性能指标

```
关键指标:

  ┌──────────────────────────────────────────────┐
  │  指标                  │  目标     │  实际范围  │
  ├──────────────────────────────────────────────┤
  │  端到端任务延迟        │  <60s    │  30-300s  │
  │  单步 TTFT (有缓存)   │  <500ms  │  200-800ms│
  │  单步 TTFT (无缓存)   │  <3s     │  1-5s     │
  │  KV 缓存命中率         │  >80%    │  50-95%   │
  │  任务完成率            │  >90%    │  60-95%   │
  │  每任务 Token 消耗     │  <50K    │  5K-200K  │
```

## 开放挑战

1. **缺乏标准 SLA**：智能体工作负载没有建立好的服务级别协议
2. **成本不可预测**：token 消耗因任务而异，差异可达 40 倍
3. **可观测性不足**：跨多步推理链的端到端追踪困难
4. **会话亲和 vs 负载均衡**：缓存亲和性与负载均衡之间的矛盾
5. **安全性**：智能体执行任意工具调用带来的安全风险
6. **GPU 利用率低**：工具执行期间 GPU 空闲
7. **长尾延迟**：少数复杂任务的延迟远超平均值
8. **评估标准**：缺乏统一的智能体服务质量评估框架
